# Chukka Score — Design Context

## Users

**Primary audiences:**

1. **Club volunteers & parent scorers** — Non-technical people at poolside, often on phones or tablets. They need to score matches quickly and accurately under pressure (wet hands, bright sunlight, crowd noise). Speed and clarity over features.
2. **Club managers** — Set up teams, players, matches, and tokens. They use the manager dashboard on laptops/desktops. They want efficiency and confidence that everything is configured correctly before match day.
3. **Spectators** — Watch live matches remotely via the public match page. They range from dedicated water polo fans to casual parents checking scores. The scorebug and timeline ARE the experience — it needs to feel like tuning into a broadcast.

**Context**: Australian water polo clubs. Matches happen at pools — outdoor and indoor. The scorer is typically a volunteer at a table beside the pool, not a professional. The spectator could be anywhere.

## Brand Personality

**Three words:** Professional. Broadcast. Trustworthy.

Chukka should feel like a real sports broadcast platform that happens to be open source and built for clubs. It's the ESPN scoreboard, not the community noticeboard. Confident, authoritative, precise — but never cold or corporate. There's warmth in serving the sport well.

**Voice & tone:**
- Direct and concise. Scorers need information, not prose.
- Data-first. Numbers, statuses, and timestamps speak louder than labels.
- Quietly confident. The interface earns trust by being accurate and responsive, not by telling you it is.

## Aesthetic Direction

**Visual tone:** Sports broadcast meets modern product design. Clean, high-contrast, data-dense where it matters (live match view), spacious where it doesn't (manager dashboard). The live match page should feel like a second-screen companion to a stream — not a web form.

**Color direction:** Aquatic blues as the brand palette. Pool water, lane ropes, the sport itself. Use blue as the primary/accent color across the product. Pair with neutral grays and high-contrast text. Reserve warm colors (amber, red) for live states, warnings, and destructive actions.

**Typography:** Instrument Sans is already in use — geometric, clean, modern. Use tabular/monospace numerals for scores, clocks, and cap numbers. Bold weights for scoreboard data. Let typography do the heavy lifting over decorative elements.

**Theme:** Light and dark mode both supported. The live match view may benefit from a dark-by-default treatment (broadcast feel), while the manager dashboard stays light-default (productivity tool).

**References:**
- FotMob (live match experience, data density, scorebug design)
- ESPN app (broadcast authority, real-time feel)
- Linear (product craft, clean layout, fast interactions)

**Anti-references (explicitly avoid):**
- **PlayHQ / TeamSnap** — amateur community-platform aesthetic, dated UI, clip-art energy, too casual
- **Betting apps** — neon colors, aggressive CTAs, dark patterns, flashy animations
- **Generic SaaS / shadcn starter kits** — it must feel like a sports product, not a dashboard template. Every screen should have a reason to exist beyond "this is how dashboards look."

## Design Principles

1. **Scoreboard first** — The live match view is the product's face. Design decisions should be tested against "does this work on the scorebug?" before anything else. If the score, clock, and status aren't instantly readable, nothing else matters.

2. **Broadcast confidence** — Every pixel should say "this data is accurate and live." Use clear status indicators, real-time feedback, and precise typography. No ambiguity about what's happening in the match.

3. **Poolside-proof** — Design for the worst conditions the scorer faces: bright sun, wet hands, split attention. High contrast, large touch targets on interactive elements, clear state changes. Errors must be obvious and recoverable (corrections, not deletions).

4. **Data density with hierarchy** — Show a lot of information without overwhelming. Use typographic scale, whitespace, and color to create clear visual hierarchy. The most important number on screen should be unmistakable.

5. **Sport, not software** — The interface should feel like it belongs in a sports broadcast, not a project management tool. Use the language of the sport (periods, quarters, exclusions, cap numbers), show team colors, and let match state drive the visual rhythm.
